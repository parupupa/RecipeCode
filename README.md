## 1. サービス概要（※最後に書いてOK）
- このサービスはどんなサービスか（3行以内）
このサービスは、レシピを保存するだけでなく、作るたびの改良内容を記録し、自分に合ったレシピとして育てていけるサービスです。

- 誰の、どんな課題を、どう解決するか
料理中に変更した点だけを簡単に記録できることで、試行錯誤を蓄積できるようにします。
日常的に料理をする人が、迷わず再現・改善できることを目的としています。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情
私はお菓子作りが好きで、同じ名前のお菓子でもいろいろなレシピを試したり、少しずつ材料や入れる量を変えて、自分の理想のレシピを作ることがあります。
いつもはスマホのメモ帳で毎回元のレシピをコピーして、その内容を書き換えて保存をしています。
何度もコピーが必要なことや、体裁が見づらくなることからエクセルなどで整理することも考えましたが、調理時にエクセルを使用するのは大変だと感じ、結局メモ帳を使っていました。
レシピの改良履歴をもっと簡単に見やすく残すことができないのだろうかと考えました。

---

### 2-2. なぜそれが「気になった」のか
私は元々料理や食べることが好きで、よく料理をします。
好きな料理は極めたいと思う性格で、作るたびに少しずつレシピを調整して、どこを調整すれば良いかを研究したり、同じ料理のレシピでもいろいろなレシピを比較してレシピを選ぶことも多いです。

これまではメモで記録残していましたが、調理中に手間がかかることやレシピが増えてきて管理が難しくなっていくことを感じ、記録はしたいが継続性や可視性に限界があることに違和感を感じたためと思います。


---

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと
レシピを書き留めておきたいが、ノートに手書きすることやPCに書き込むのは料理中は面倒。スマホの場合もメモ帳だとカーソルの操作などが細かくてメモを取るのが大変。
---

### 3-2. 本当に解決したい課題は何か
- なぜ？
すぐにメモできないので、レシピを記録する機会を逃す

- なぜ？
手近な紙や謎のファイルなどに走り書きしてしまうことが起き、結局どこに書いたかわからなくなる。

- なぜ？
書いた媒体やフォーマットがバラバラで、何が一番良いレシピだったかわからなくなる

→ 本当に解決したい課題：レシピの改良履歴を手軽に記録して、見やすく探しやすく整理したい

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

年齢設定：
  - 年齢：20代〜50代
  - 理由：自炊が習慣化しており、レシピを「探す」だけでなく「自分なりに調整する」機会が多い年代のため

生活・状況設定：
  - 生活・状況：自炊をすることが多い、料理やお菓子作りが趣味
  - 理由：  自分で作ったレシピを保存しておきたい時や、少しレシピを調整してそれをメモしておきたいとき、手元にあるスマホで簡単に記録できる。  後からまた同じレシピを見たときに、レシピの調整がしやすくなる。

環境設定：
  - 環境：スマートフォンでレシピを見ながら料理をする
  - 理由：料理中でも簡単に記録できるUIにすることで、記録のハードルを下げる

---

### 4-2. 自分とユーザーの距離
- 自分自身

  検証方法としては、本アプリが自分自身の課題から生まれているため、まずは自分で使いながら改善していくことを想定している
  その上で、同様に日常的に料理をしている友人や家族に試してもらい、操作のしやすさや記録のしやすさについてフィードバックを得ることで、より多くのユーザーにとって使いやすい形に改善していきたいと考えている。

## 9. 今後の展開・発展の方向性
### 観点（すべて答えなくてOK）

- ユーザーはなぜ使い続ける？
  → 料理をするたびに改良履歴が蓄積され、自分に合ったレシピが完成していくため
  → 一度使うほど「次も使った方が楽になる」という状態になるため

- 続けることで何が変わる？
  → 毎回試行錯誤する必要がなくなり、料理の再現性が高まる
  → 自分の好みや傾向が明確になり、迷わず料理できるようになる

- データが溜まると何ができる？
  → よく使う調整や好みの傾向を可視化できる
  → 自分にとって最適なレシピを一覧で把握できる
  → 「自分専用のレシピ集」として機能する

- 他の人と関わる余地はある？
  → 将来的には、自分の改良履歴を他のユーザーと共有し、他人の工夫を参考にできる可能性がある
  → ただしMVPでは個人利用に特化し、無理にコミュニケーション機能は持たせない

- 運用・仕組みで価値を出せないか？
  → 記録を強制するのではなく、自然に続けられる設計にすることで、継続的な利用を促す
  → 「変更部分だけ入力」という仕組みにより、記録のハードルを下げる

### 今考えている発展案

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術
- フレームワーク：Ruby on Rails(7.0系)
- 言語 Ruby(3.1.4)
- DB：PostgreSQL
- デプロイ先：Render
- 使用予定ライブラリ：Devise/ Ransack / ActiveStorage / TailwindCSS

---

### 10-2. 技術選定の理由
Ruby on Rails：
CRUD処理に強く、効率よく開発できるため。
本アプリのようなデータ管理中心の構成に適している。

PostgreSQL：
Railsとの相性が良く、信頼性の高いデータ管理が可能なため。

Render：
無料プランがあり個人開発でも扱いやすく、Railsアプリのデプロイも比較的簡単に行えるとの調査により選定。

Devise：
当初はSorceryの利用を検討していたが、長期間更新が行われていない点を踏まえ、  
継続的にメンテナンスされているDeviseの方が適していると判断し採用した。

Ransack：
検索機能の実装のため。

ActiveStorage：
Rails標準の機能であり、追加の設定が少なく扱いやすいため。

TailwindCSS：
シンプルで見やすい画面を効率的に構築でき、料理中でも迷わず操作できるインターフェースを実現しやすいため。

- 今回チャレンジしたい点
変更部分のみ入力し、保存時に完成形を生成する改良履歴の設計に挑戦したい。
また、元レシピ・最新版・履歴を分かりやすく表示するUI設計にも取り組む

- 不安な点
改良履歴の設計が複雑になりすぎないか不安がある。
また、機能を増やしすぎず、シンプルなUIを維持できるかが課題ではないかと思う。

### 本サービスの概要（700文字以内）
このサービスは、レシピを保存するだけでなく、料理をするたびに加えた工夫や調整内容を記録し、改良履歴として蓄積できるアプリです。
ユーザーは変更した部分のみを簡単に入力するだけで、その時点の完成形レシピが自動的に保存され、過去の試行錯誤を振り返ることができます。
これにより、毎回同じ調整を繰り返す手間や、記録の分散による管理の煩雑さを解消し、自分に合ったレシピを効率よく再現・改善できるようになります。
料理中でも使いやすいシンプルな操作性を重視し、継続的に記録しやすい設計とすることで、日常の調理をより快適で前向きな体験に変えることを目指します。

### MVPで実装する予定の機能
- ユーザー登録（名前・メールアドレス・パスワード）
- ログイン・ログアウト
- パスワードリセット（メール送信・再設定）
- 登録情報変更（名前・メールアドレス・パスワードをそれぞれ個別に変更）
- アカウント削除
- レシピ作成（タイトル・材料・手順・画像）
- レシピ一覧表示（ページネーション・並び替え）
- レシピ検索（Ransackによるタイトル部分一致）
- レシピ詳細表示（最新バージョンを優先表示）
- レシピ削除
- 改良記録作成（変更箇所のみ入力・元バージョンの内容を引き継ぎ・メモ・画像添付）
- 改良履歴一覧表示（並び替え・最新版の強調表示）
- 改良履歴詳細表示
- 改良履歴削除


### テーブル詳細
#### users
- id : integer / 主キー。自動採番
- name : string / ユーザーの表示名。登録・マイページに表示（必須）
- email : string / ログイン用メールアドレス（必須・ユニーク制約）
- encrypted_password : string / Deviseが管理するハッシュ化済みパスワード
- reset_password_token : string / パスワードリセット用の一時トークン（Devise管理）
- reset_password_sent_at : datetime / パスワードリセットメールの送信日時。有効期限の判定に使用（Devise管理）
- created_at : datetime / レコード作成日時（Rails自動管理）
- updated_at : datetime / レコード更新日時（Rails自動管理）

#### recipes
- id : integer / 主キー。自動採番
- user_id : integer / 外部キー。作成したユーザーのid（必須）
- title : string / レシピ名（必須）
- ingredients : text / 元レシピの材料。改良前のベースとして参照される
- steps : text / 元レシピの手順。改良前のベースとして参照される
- created_at : datetime / レコード作成日時（Rails自動管理）
- updated_at : datetime / レコード更新日時（Rails自動管理）

#### recipe_versions
- id : integer / 主キー。自動採番
- recipe_id : integer / 外部キー。紐づくレシピのid（必須）
- version_name : string / ユーザーが任意でつけるバージョン名。自動採番の番号が初期値
- version_number : integer / レシピごとの通し番号。0 = 元レシピ、1以降 = 改良履歴
- ingredients : text / このバージョンの材料全文
- steps : text / このバージョンの手順全文
- memo : text / 今回の改良で試したこと・気づきのメモ（任意）
- image : string / ActiveStorageの添付ファイル識別子
- created_at : datetime / レコード作成日時（Rails自動管理）
- updated_at : datetime / レコード更新日時（Rails自動管理）
